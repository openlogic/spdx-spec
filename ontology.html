2 Vocabulary

This section defines the vocabulary used to describe packages, licenses, copyrights and other information in SPDX. This vocabulary represents the abstract model of SPDX.

2.1 Classes

An SPDX document is composed of a set of resources. A resource is an entity such a file, license or package. Each resources has a class. The class of a resource lets SPDX consumer know what sort of resource it is.

This is a list of the types of entities defined by SPDX, and what properties are associated with resources of these types.

2.1.1 SPDXDoc

An SpdxDoc is a collection of license and copyright information regarding one or more packages and the files contained in those packages.

Properties

  • SPDXVersion
    Cardinality: Mandatory, one (1).

  • CreatedBy
    Cardinality: Mandatory, one (1) or more.

  • Created
    Cardinality: Mandatory, one (1).

  • ReviewedBy
    Cardinality: Optional, zero (0) or more.

2.1.2 Package

A package is an artifact of a software development effort. It is generally a snapshot of the source code at a particular point in time or an mechanism to be used to install the software. A package can contain sub-packages, but the overview is a reference to the entire contents of the package listed.

Properties

  • DelcaredName
    Cardinality: Mandatory, one (1).

  • MachineName
    Cardinality: Mandatory, one (1).

  • SHA1
    Cardinality: Mandatory, one (1).

  • DownloadURL
    Cardinality: Optional, one (1).

  • SourceInfo
    Cardinality: Optional, one (1).

  • DelcaredLicense
    Cardinality: Optional, zero (0) or more.

  • DetectedLicense
    Cardinality: Optional, zero (0) or more.

  • DeclaredCopyright
    Cardinality: Optional, zero (0) or more.

  • ShortDesc
    Cardinality: Optional, one (1).

  • Description
    Cardinality: Optional, one (1).

2.2 Properties

Each resource in an SPDX file has as set of properties which describes that item. The class also defines what properties are allowed for that resource. The value of a property may be a literal value or a resource of a specific class.

This section provides a details description of the various properties use to convey package data. Some of the properties apply to more than one class of resource.

2.2.1 SPDXVersion

This property applies to SPDXDoc and provide a reference number that can be used to understand how to parse and interpret the rest of the file. It will enable both future changes to the specification and to support backward compatibility.

The version identifier consists a format identifier, a major and a minor version indicator. The Major field will be incremented when incompatible changes are made to the specification. The Minor field will be incremented when any change is made that is fully backwards compatibility. An SPDX processor may safe read a file that has the expected format identifier and major version.

The value of the SPDXVersion property for all datasets encoded in this verison must be: SPDX-1.0

2.2.2 CreatedBy

This property identifies how information in the SPDXDoc was generated. The value of CreatedBy is an xsd:string. The format of this string may indicate a person, tool or organization.


      company    = "Company:" *WSP name *WSP ["(" email ")"]
      tool       = "Tool:" *WSP name
      person     = "Person:" *WSP name *WSP ["(" email ")"]
      name       = *VUNICODECHAR
      email      = 1*(ALPHA / DIGIT / "." / "-" / "_") "@" domainpart 1*("." domainpart)
      domainpart = 1*(ALPHA / DIGIT / "-" / "_")

For example, "Person: John Doe (john.doe@example.com)" or "Company: My Company, Inc" (depending on which is accountable) or "Tool: spdx-gen 1.0".

2.2.3 Created

This property identifies when the analysis was done and the SPDXDoc was originally created. The value of this property is an xsd:dateTime

This property refines the definition of dc:created.

2.2.4 ReviewedBy

This property allows the recording of any reviews that have been done of a SPDXDoc. The value of this property is an xsd:string containing the reviews name and, optionally, email address and the date on which the review took place. 


      review = "person *WSP timestamp"

For example, "Person: John Doe (john.doe@example.com) 2010-11-02T17:02:03Z".

Note that there is no requirement for a particular reviewer to add their name to the file, however it may be important for participants in the software supply chain to validate whether upstream providers have reviewed the SPDX file. This can be considered as an equivalent to "signed off" or "reviewed by". Additional reviewers can be added after the original version of the SPDX file is created and can occur as an append to the original file.

2.2.5 DeclaredName

This property records the name and version of a Package. The value is an xsd:string that contains the full name, including version information if known, of the package. This property in intended for primarily human consumption so the arrangement of this information is not formally defined.

2.2.6 MachineName

This property stores the file name of the Package. The value is an xsd:string that is the machine generated file name and version typically includes extensions to indicated the packaging and compression methods used.

For example, "zlib-1.2.5.tar.gz".

2.2.7 SHA1

This property provides an independently reproducible mechanism that permits verification that a specific Package or File correlates to the information in this SPDX file. This identifier enables a recipient to determine if any file in the original package has been changed. The value is the hexadecimal encoded SHA1 hash of the package or file in an xsd:string. The length of this values is always 40 characters.
